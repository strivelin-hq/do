Development & Deployment Workflow

A step-by-step breakdown of scripts, dockerization, database versioning, and parity comparison.

01

Code Compilation ./scripts/build.sh

Compiles the Next.js React and TypeScript codebase locally. This step is executed to identify any type mismatches, TypeScript syntax errors, or compiler warnings prior to packaging the application.

$ npm run build
# Executes: tsc -b && next build
02

Image Packaging ./scripts/stage.sh

Assembles all necessary environment configuration files and packages the Next.js application into a production-ready, lightweight Docker image.

$ docker compose build
# Packages Next.js with a multi-stage release target to save space.
03

Deployment & Auto-Migration ./scripts/run.sh

Spins up the database container (PostgreSQL) and the application container. Once the database is online, the auto-migration script runs, executing any new sql migrations and updating the schema version table.

$ docker compose up -d
$ node scripts/migrate.js
# Connects, checks `schema_migrations`, runs new SQL scripts, inserts new version.
04

Teardown & Cleanup ./scripts/stop.sh

Safely stops all running containers, closes database connections, and clears the virtual environment resources while keeping local database volumes intact.

$ docker compose down
# Shuts down containers, keeps database volumes preserved in Docker storage.
05

Schema Parity Check ./scripts/compare.sh

Ensures database migrations are completely clean. It initializes two temporary databases in Docker: a fresh install (sequential migrations on an empty DB) and a migrated install (migrations applied onto an existing setup). It runs a schema comparison tool to detect columns, types, or policy drift.

$ docker compose -f docker-compose.compare.yml up -d
$ node scripts/compare-schemas.js
# Queries catalogs, checks Tables, Columns, Constraints, and RLS policies.
v

Version Control in the Database

To prevent database drift and ensure safe cloud deployments, the database contains a table named schema_migrations. Every time scripts/migrate.js runs, it checks this table to determine which migration scripts (e.g. 0001_create_goals_table.sql, 0002_add_target_date_recurrence.sql) have already been applied, ensuring database updates are sequential, predictable, and fully tracked.
